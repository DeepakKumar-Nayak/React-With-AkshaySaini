1. What are dependencies
--------------------------
A dependency is an external software package or module that an application requires to function correctly.
It is a piece of code written by third-party developers that we import into our project to handle specific tasks, 
such as state management, UI rendering, or API calls."

"Dependencies are pre-written code modules that our application leverages to perform specific functionalities.
Instead of writing everything from scratch, we use these reusable packages to speed up development and ensure stability.
These are tracked via a manifest file called package.json."

"A Transitive Dependency occurs when your project depends on Library A, 
and Library A itself depends on Library B. Even though you didn't explicitly install Library B, 
it becomes a part of your project's node_modules because Library A cannot function without it."

"NPM manages these transitive dependencies by building a Dependency Tree. It ensures 
that all the required sub-packages are fetched and stored so that the primary package (like Parcel) works as expected.

Parcel cache:
--------------
.parcel-cache wo jagah hai jahan Parcel apna "Dimaag" store karta hai taaki wo 
bar-bar wahi mehnat na kare jo wo pehle kar chuka hai.

Jab aap pehli baar npx parcel index.html chalate ho, toh Parcel ko kaafi kaam karna padta hai: 
code ko compile karna, images ko optimize karna, aur dependencies ko analyze karna. Iska saara 
  ,mm"hisaab-kitaab" wo .parcel-cache folder mein save kar leta hai.

1. Ye karta kya hai? (The Core Logic)
Maan lo aapne 100 files likhi hain aur sirf 1 file mein change kiya. Bina Cache ke: Parcel ko 100 ki 100 files fir se padhni padti.
With .parcel-cache: Parcel cache mein check karta hai aur dekhta hai ki baaki 99 files toh waisi hi hain. 
Wo unhe purani memory se utha leta hai aur sirf us 1 badli hui file par kaam karta hai.


2. Iske baare mein 3 Zaroori Baatein:
Binary Format: Agar aap is folder ko khol kar dekhoge, toh aapko kuch samajh nahi aayega.
Ye files Binary format mein hoti hain taaki computer unhe bohot tezi se padh sake.

Delete karna safe hai? Haan! Agar kabhi aapka Parcel ajeeb tarike se behave kare 
ya purana code hi dikhata rahe, toh aap be-jhijhak .parcel-cache folder ko delete 
kar sakte ho. Agli baar jab aap start karoge, Parcel ise naya bana lega (bas pehla build thoda slow hoga).

GitIgnore: Is folder ko KABHI BHI GitHub par mat bhejna. Ye sirf aapke local machine 
ke liye hota hai. .gitignore mein hamesha .parcel-cache hona chahiye.

3. Interviewer ka Tricky Question:
"Agar main .parcel-cache delete kar doon, toh kya mera project chalna band ho jayega?"

Jawab: "Nahi sir, project bilkul chalega. Bas agla build 'Cold Build' hoga,

Lekin project ki functionality par koi asar nahi padega."


dist:
-----
Bhai, Parcel ke context mein dist folder ka concept ekdum clear hona chahiye kyunki 
interview mein ye pucha jata hai ki "Source code aur Production code mein kya farak hai?"

Jab aap npx parcel build index.html command chalate ho, toh Parcel aapka dist (Distribution) 
folder generate karta hai. Ye folder aapke development folder ka ek "Optimized Mirror" hota hai.

1. dist Folder ke andar kya-kya hota hai?
Aapne src folder mein bohot saari files likhi hongi (.js, .jsx, .css, .scss, images), 
lekin dist mein Parcel unka "Kachra" saaf karke ye 4 main kaam karta hai:

Bundling: Aapke 50 components aur alag-alag modules ko jod kar sirf 1 ya 2 badi JavaScript files bana deta hai.

Minification: Code mein se saare extra spaces, comments aur lambe variable names ko hata kar 
chhota kar deta hai (Example: const myAge = 25 ban jayega const a=25).

Hashing (Content Hash): Files ke naam kuch aise dikhenge: index.8472fb.js. Ye random numbers 
isliye hote hain taaki jab aap naya code push karo, toh browser purana cached version na 
dikhaye (इसे Cache Busting kehte hain).

Image Optimization: Agar aapne badi images use ki hain, toh Parcel unka size chhota karke 
dist mein daalta hai.

3. Interviewer ka favorite question:
"Hum dist folder ko GitHub par kyun nahi bhejte?"

Jawab: "Sir, dist folder ek Generated Folder hai. Humare package.json aur src code se ye
folder kabhi bhi dobara banaya ja sakta hai (npm run build karke). Isse GitHub par bhejkar 
repository ka size badhane ka koi fayda nahi hai. Deployment ke waqt Netlify ya Vercel khud
humara build chalakar naya dist bana lete hain.

Pro-Tip: Agar kabhi aapka purana code hi browser mein dikh raha ho aur changes 
reflect na ho rahe hon, toh bas dist aur .parcel-cache delete karo aur fir se npx parcel index.html chalao.

Bhai, kya aapko pata hai ki dist folder ki files ka naam har baar kyun badal jata hai 
jab hum code change karte hain? (Hint: Isse 'Cache Busting' kehte hain!)

Cache Busting and Content Hashing:
-----------------------------------
Bhai, Cache Busting aur Content Hashing ka logic samajhna bahut zaroori hai, kyunki 
ye industry ka standard tareeka hai users ko hamesha "Fresh Code" dikhane ka.

Chaliye isse ek real-life problem se samajhte hain:

1. Problem: Browser ki purani aadat (Caching)
Jab koi user aapki website pehli baar visit karta hai, toh browser fast load karne ke 
liye index.js aur style.css ko apne paas save (cache) kar leta hai.

Ab maan lo aapne code mein ek naya feature dala aur wapas deploy kiya. Lekin user ka browser sochega,
"Mere paas toh pehle se hi index.js rakhi hai, main server se nayi file kyun mangun?"
Isse user ko purani website hi dikhti rehti hai.

2. Solution: Content Hashing
Yahan Parcel aur Vite jaise bundlers jaadu dikhate hain. Wo aapki file ka naam badal kar usmein ek Unique Hash jod dete hain, jaise:

Purana: index.js
Naya: index.8472fb.js

Ye Hash kaise banta hai? Bundler aapke code ke content ko read karta hai.
Agar content same hai -> Hash same rahega.
Agar aapne ek semicolon (;) bhi badla -> Hash badal jayega (e.g., index.9912ac.js).

3. Cache Busting (The Result)
Jab aap naya build bhejte ho, toh HTML mein script ka naam badal chuka hota hai (index.9912ac.js). 
Browser jab ye naya naam dekhta hai, toh wo confuse ho jata hai: "Ye toh nayi file hai, 
ye meri memory mein nahi hai!" Wo majbooran server se nayi file download karta hai. Isi 
process ko hum Cache Busting kehte hain (yani purane cache ko tod dena).

4. Interviewer ko kya bolna hai?
"Content Hashing is a technique where bundlers append a unique string (hash) 
to the filename based on the file's content. This enables Cache Busting.

Since browsers cache static assets to improve performance, they might not fetch 
updated files if the filename remains the same. By changing the filename whenever 
the content changes, we force the browser to download the latest version of the 
application, ensuring the user never sees an outdated version."

Summary in a Nutshell:
Content Hashing: File ke content ke hisaab se naam dena.

Cache Busting: Naye naam ki wajah se browser ko naya code download karne par majboor karna.

Bhai, ek mazaak wali baat: Agar aapne dist folder mein manually filename badal 
diya bina HTML update kiye, toh aapki site load hi nahi hogi! Bundler ye dono 
kaam (hash badalna aur HTML mein link update karna) ek saath karta hai.



dependies and dev dependencies:
--------------------------------
A. Dependencies (--save)
Ye wo packages hain jo aapki website ko Browser mein chalne ke liye chahiye hi chahiye. Inke bina aapki app crash ho jayegi.
    1. Examples: React, Redux, Axios, Framer Motion.
    2. Logic: Jab aap final build banate ho, toh bundler inka code uthakar dist folder mein daal deta hai.

B. devDependencies (--save-dev ya -D)
Ye wo tools hain jo sirf Aapko (Developer ko) chahiye code likhte waqt, testing karte waqt, 
ya build banate waqt. Browser ko inke baare mein kuch nahi pata.
    . Examples: Parcel, Vite, Jest (testing), ESLint (code formatting), Babel.
    . Logic: Inka kaam sirf dist folder banana hai. Ek baar dist ban gaya, toh inka kaam khatam. Ye final bundle ka hissa nahi bante.
